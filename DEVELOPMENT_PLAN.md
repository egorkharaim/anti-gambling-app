# План разработки Bankroll Discipline Manager

## 1. Границы первой версии

### Приложение должно

- хранить один локальный финансовый профиль;
- рассчитывать свободный доход и рекомендуемый игровой лимит;
- создавать месячные и недельные планы;
- разрешать только уменьшение активного лимита;
- учитывать игровые сессии и их финансовый результат;
- отдельно показывать чистый результат и использованный лимит потерь;
- фиксировать нарушения правил;
- показывать dashboard, календарь дисциплины и базовую статистику;
- работать как web-приложение на Java с HTML/CSS/JavaScript.

### Приложение не должно в MVP

- регистрировать и авторизовывать пользователей;
- блокировать сайты или приложения;
- хранить данные в облаке;
- принимать или переводить реальные деньги;
- подключаться к казино или букмекерским сервисам;
- рекомендовать ставки;
- прогнозировать выигрыши;
- содержать достижения и игровую геймификацию.

## 2. Нефункциональные требования

- Все финансовые правила детерминированы и покрыты тестами.
- Денежные значения представлены типом `Money`, который содержит `BigDecimal`
  и код валюты.
- Scale суммы определяется валютой: например, для PLN допустимо два знака после
  запятой. Значение, требующее округления, отклоняется; лишние завершающие нули
  безопасно нормализуются.
- `double` и `float` запрещены для денег и процентов. Также запрещен конструктор
  `new BigDecimal(double)`; значения создаются из строк или целых чисел.
- Любое деление и округление выполняется с явно указанным `RoundingMode`.
- Перед сравнением и сохранением суммы приводятся к каноническому scale валюты.
- В MySQL суммы хранятся в точных колонках `DECIMAL(19,4)`, а не в `FLOAT` или
  `DOUBLE`. После чтения Java-адаптер проверяет и нормализует scale по валюте.
- Время хранится как `Instant` в UTC, а отображается в выбранной временной зоне.
- Изменение профиля влияет только на будущие планы.
- Исторические планы и сессии не пересчитываются задним числом.
- Миграции базы данных применяются автоматически и имеют версию.
- Локальная MySQL запускается через Docker Compose; пароли не хранятся в Git.
- Интеграционные тесты используют MySQL через Testcontainers, а не H2.
- `mvn clean verify` должен проходить локально и в GitHub Actions.
- Для критических доменных правил требуется полное покрытие ветвей тестами; для
  остального собственного кода целевой уровень — не менее 70%.

## 3. Архитектура

Проект создается как Maven multi-module приложение:

```text
bankroll-manager
├── bankroll-domain
│   └── расчётное ядро: деньги, профили, планы, сессии и чистые правила
├── bankroll-application
│   └── сценарии использования и интерфейсы репозиториев
├── bankroll-infrastructure
│   └── MySQL, Liquibase и реализации репозиториев
└── bankroll-web
    └── Spring Boot, REST-контроллеры, HTML, CSS и JavaScript
```

Зависимости направлены внутрь:

```text
web ───────► application ───────► domain
infrastructure ────────────────► application/domain
```

`bankroll-domain` — это независимое расчётное ядро. Оно не зависит от Spring,
базы данных или web-слоя: здесь находятся только модели и правила предметной
области. Это не отдельный сервер и не часть интерфейса. Такое разделение не дает
контроллерам или JavaScript самостоятельно пересчитывать лимиты и позволит позже
использовать те же правила при разработке мобильного приложения.

## 4. Зафиксированные финансовые правила

### 4.1 Свободный доход

```text
protectedMoney = mandatoryExpenses + savingsGoal + emergencyContribution
disposableIncome = max(monthlyIncome - protectedMoney, 0)
recommendedLimit = disposableIncome * selectedPercentage / 100
```

Если защищенные расходы выше дохода, рекомендуемый лимит равен нулю, а результат
содержит предупреждение о дефиците.

### 4.2 План периода

- План имеет начальный и текущий лимит.
- После активации текущий лимит не может стать больше предыдущего.
- Попытка повышения отклоняется доменным правилом, а не только интерфейсом.
- Уменьшить лимит ниже уже использованной суммы можно. В таком случае план сразу
  получает состояние `EXCEEDED`; история не изменяется.
- Периоды одинакового типа не могут пересекаться.
- Изменение финансового профиля не изменяет уже созданный план.

### 4.3 Использование лимита потерь

Для каждой завершенной сессии вычисляется чистый результат:

```text
sessionResult = closingBalance + withdrawn - openingBalance - deposited
```

Использованный лимит потерь:

```text
sessionLoss = max(-sessionResult, 0)
usedLossLimit = sum(sessionLoss for all sessions in period)
```

Положительный результат одной сессии не компенсирует убыток другой сессии внутри
активного периода. Чистый результат периода показывается отдельно.

### 4.4 Сессии

- Одновременно может существовать только одна активная сессия.
- Время завершения не может быть раньше времени начала.
- Завершенная сессия не может быть завершена повторно.
- Финансовые поля не могут быть отрицательными.
- Оборот ставок является необязательной статистикой и не участвует в расчете
  лимита потерь.

## 5. Основные модели

- `id: Long`
- `currency: CurrencyCode`
- `monthlyIncome: Money`

- `savingsGoal: Money`
- `emergencyContribution: Money`
- `allocationPercentage: Percentage`
- `timeZone: ZoneId`
- `createdAt: Instant`
- `updatedAt: Instant`
- `version: long`

### BankrollPlan

- `id: Long`
- `type: MONTHLY | WEEKLY`
- `periodStart: LocalDate`
- `periodEnd: LocalDate`
- `recommendedLimit: Money`
- `initialLimit: Money`
- `currentLimit: Money`
- `sessionLossLimit: Money`
- `maxSessionsPerDay: int`
- `maxSessionDuration: Duration`
- `status: DRAFT | ACTIVE | COMPLETED | EXCEEDED`
- snapshot финансовых параметров, по которым создавался план;
- `createdAt: Instant`
- `version: long`

### LimitAdjustment

- `id: Long`
- `planId: Long`
- `previousLimit: Money`
- `newLimit: Money`
- `reason: String`
- `createdAt: Instant`

### GameSession

- `id: Long`
- `startedAt: Instant`
- `endedAt: Instant?`
- `openingBalance: Money`
- `deposited: Money`
- `withdrawn: Money`
- `closingBalance: Money`
- `turnover: Money?`
- `result: Money`
- `lossLimitUsed: Money`
- `notes: String?`
- `status: ACTIVE | COMPLETED`
- `version: long`

### RuleViolation

- `id: Long`
- `planId: Long?`
- `sessionId: Long?`
- `type: ViolationType`
- `severity: INFO | WARNING | EXCEEDED`
- `actualValue: String`
- `limitValue: String`

## 6. Web-страницы

- `/` — dashboard активных лимитов и предупреждений;
- `/profile` — финансовый профиль;
- `/plans` — список и создание планов;
- `/plans/{id}` — детали и уменьшение лимита;
- `/sessions` — история сессий;
- `/sessions/active` — активная сессия;
- `/calendar` — календарь дисциплины;
- `/statistics` — статистика по периодам.

Интерфейс должен напоминать финансовый или habit-tracking продукт. Казино-стиль,
мигающие элементы, праздничная анимация выигрышей и стимулирующие формулировки
запрещены.

## 7. API первой версии

### Profile Controller

- `GET /api/profile` — получить профиль;
- `PUT /api/profile` — создать или изменить параметры профиля.

После создания первого плана `currency` изменять нельзя. Остальные изменения
используются только в расчетах будущих планов.

### Plans Controller

- `POST /api/plans` — создать план;
- `GET /api/plans` — получить список планов;
- `GET /api/plans/{id}` — получить план и его использование;
- `PATCH /api/plans/{id}/limit` — только уменьшить активный лимит;
- `POST /api/plans/{id}/activate` — активировать draft-план.

### Sessions Controller

- `POST /api/sessions/start` — начать сессию;
- `POST /api/sessions/{id}/complete` — завершить сессию;
- `GET /api/sessions` — получить историю с фильтрами по датам;
- `GET /api/sessions/{id}` — получить детали сессии.

### Dashboard and Reports Controller

- `GET /api/dashboard` — активные планы, остатки и предупреждения;
- `GET /api/calendar?month=YYYY-MM` — статусы дней;
- `GET /api/statistics?from=...&to=...` — агрегированная статистика;
- `GET /api/violations` — список зафиксированных нарушений.

Для совместных задач создается один PR с одним назначенным автором, а не два
конкурирующих PR. Перед началом задачи разработчики фиксируют DTO и интерфейсы use
case. После этого каждый работает только в своей части (`domain/application/data`
или `web`).

### Рекомендуемый порядок параллельной работы

1. Оба разработчика совместно завершают PR 1.
2. Разработчик 1 последовательно выполняет PR 2–5. В это время разработчик 2
   готовит статические прототипы profile, plans и sessions без собственной
   финансовой логики.
3. PR 6 и PR 7 выполняются совместно после стабилизации profile и plan API.
4. Разработчик 1 выполняет PR 8; разработчик 2 готовит UI сессии на согласованных
   mock-данных.
5. После PR 9 разработчик 1 берет PR 10, а разработчик 2 начинает PR 11 только
   после появления dashboard use case.
6. PR 12 и PR 13 делятся по таблице выше; PR 14 проверяется обоими разработчиками.

Ни один расчет денег, лимитов, результатов сессий или статусов календаря не
дублируется в JavaScript. Web-слой только отправляет данные и отображает результат
Java use case.

## 10. Вопросы, которые нужно закрыть до соответствующих PR

Эти решения не следует угадывать заранее:

1. С какой даты начинается неделя: понедельник или пользовательская настройка?
2. Может ли недельный план существовать без месячного или всегда является его
   частью?
3. Какой порог означает «слишком быстрое расходование» лимита?
4. Какой интервал после проигрыша считается возможной погоней за потерями?
5. Можно ли исправлять ошибочно завершенную сессию и каким образом сохранять
   историю исправления?
6. Разрешается ли выбранный лимит будущего периода выше рекомендации или это
   только фиксируется как предупреждение?

До ответа соответствующая логика не реализуется и не маскируется случайными
значениями по умолчанию.
